Known Bugs

After several design processes and testing, our application is almost without bugs. The only one that is left is of a rather logical nature. This means that in no manner is this going to affect the user productivity while making use of our application. However, there is still one drawbacks to consider:

Bug Description Priority
Exposure: when a user logs in into our application, some of his data is stored into a json file which is within the internal memory of the phone. The same file is sent to a RESTful API that processes the information and allows the user to go over and use a browser to visualise some of his game data using a specific link. However, whenever another user logs in using the same phone, some of his data is stored into the same json file. This means that the data of the users is exposed through the entire API, which could lead to data privacy concerns in the future. High

Critical Evaluation of Project

User Interface

We have designed our use interface with an accessible menu so the user can navigate to any page with ease. In this way the application is convenient to use and isn't time consuming for the user to retrieve any information. Additionally, the interface uses a consistent colour theme and icons to make the application to both make the application more aesthetically pleasing and also ensure that it is straightforward to use.

Once the user has entered the main dashboard, the interface displays the data as colour coded graphs, representing data from different games. In this way, it is easier for the user to compare their statistics between different games.

The application is scrollable, and users are also able to scroll horizantally through their different statistics or vertically through different games. This makes the navigation of the application smooth and more pleasing to use.

Functionality

Although at present the application uses replica data, the front end meets all of the functional 'must have' requirements, and even some 'should haves' and 'could haves'. Within the scope of our project, it was not possible to implement these features as a part of a fully functional application, but with further development on the back end in terms of retrieving data, the application will meet all the requirements.

Stability

To make our code stable, we produced extensive unit tests. If the tests are repeatedly run when the code is changed, then it is easy to ensure all functionalities still work.

Efficiency

To ensure our code is efficient, we complied with object-oriented best practices in terms of the structure of our code, using correct classes and design patterns such as adapters and builders.

Compatibility

At present, our application is only compatible on Android devices. In terms of our requirements, this fulfills the 'must have' that the application must compatible on either Android or iOS. This was decided as the result from our surveys which showed half of the current potential end users use iOS devices and half of them use Android devices.

However, this discounts half of our end users and therefore as a 'should have' requirement we decided the application should be compatible on both types of devices. Given the limited time frame we had, this was not possible, but will definitely be a next step for further development of the application.

Maintainability

Our code is structured is such a way to ease the maintenance for future developers. For the front end, we have used classes to separate each screen so it is easy to identify what section of the code needs modifying if a future developer wants to change an element of specific screen. Additioanlly, as mentioned in the design section, we have used adapters which allow modifications to specific elements without affecting the functionality of other parts of the code.

In terms of the code itself, it is commented throughout for future developers to edit the legacy code with ease. Variable names have been chosen to ensure the code is self-explanatory.

Project Management

Our team was tested throughout the development process due to many unexpected issues and resulting changes. For this reason, it was a difficult task to maintain the management of our project and despite the difficulties we faced, we managed to keep up with each change by recognising each team member's strengths and dividing the tasks accordingly.

Although a lot of our development process was spent not knowing what our client wanted, we all took the time to research different technologies and data analysis techniques so that we would have the knowledge to take on what our client wanted from us.

What we have learnt from the way we approached project management is to get a clear set of requirements from our client from the beginning, and not allow them to change these requirements throughout the process. In this way, we will have a consistent targets throughout, and will not make the task harder for ourselves.

Future Work

With our project impacted by major setbacks, our final deliverable is designed to be the start of further development. For this reason there is plenty of scope for future work. Looking into the next three months, we would hope to achieve the following:

Month 1
Within the first month, we would hope to retrieve data from the Fitbit API and integrate this with the mobile application. In this way, we will use real data, allowing the application to actually be used alongside a wristband containing a sensor. The application will be functional for an end user at this point, instead of just being a stepping stone for our client, a further developer.

Month 2
In the second month, we will make our application compatible both with iOS and Android. This will have a significant impact on the range of end users our application will target; the application will be usable by the entire set of current end users as opposed to half of them. In order to implement this, we will develop the front end using Xamarin. From the first month, we will have the back end retrieving data from the Fitbit API, which we will also integrate with out iOS application at this point.

Month 3
Looking further into the future, we would hope to extend the functionality of the application. One feature we looked into would be to link users' accounts and create some sort of leader board between team members so that they could see better how they perform as a team and individually. We also considered further extending this to creating a separate log in system for a coach, so they could see the statistics of all players.
